# Red-Team / Blue-Team Security Assessment — 2026-06-29

Authorized internal security exercise on the WorkUnit OS codebase. Four
attack subagents (red team) probed distinct surfaces in an isolated git-worktree
sandbox; the blue team (main) triaged the findings and shipped tested hardening
patches. No production/main code was modified during the exercise — all work
happened in the `ai_editor_sandbox_redteam` worktree on branch
`security/redteam-hardening`.

## Methodology

- **Sandbox**: dedicated git worktree seeded from the live working tree
  (1827-test green baseline at commit `799d472`). `node_modules` shared
  read-only. Red-team agents were read-only on source; PoCs ran from scratchpad.
- **Static verification**: source review of auth, RBAC, CSRF, routes, the LLM
  pipeline, persistence, crypto, and the Electron shell.
- **Dynamic verification**: each attacker wrote runnable PoCs
  (`node --experimental-strip-types`) to confirm exploitability; the blue team
  added a regression suite (`tests/redteamHardeningP0.test.mts`) that fails
  without the patch and passes with it.

## Overall security posture

**Level: B (good baseline, fixable gaps).** The architecture gets the hard
parts right and these held up under attack:

- JWT verification rejects `alg:none`/algorithm confusion; `exp`/`nbf` validated
  with `Number.isSafeInteger`.
- All **D1** queries are parameterized — no SQL injection; tenant scoping is
  correct in every `WHERE` clause.
- One-time approval use is an atomic compare-and-set; cross-tenant approval
  replay is blocked by `verifyApproval` + tenant scoping.
- CSRF origin validation is enforced on every state-changing POST.
- HMAC comparisons use `timingSafeEqual`.
- Electron shell: `nodeIntegration:false`, `contextIsolation:true`,
  `sandbox:true`, IPC restricted to two read-only channels.
- `safeErrors` prevents stack/detail/secret leakage in responses.

The weaknesses clustered in **input canonicalization** (every pattern guard was
ASCII-only and fell to Unicode homoglyphs), **defense-in-depth** (in-memory
repos lacked the tenant scoping their D1 counterparts enforce), and **a few
missing validations** (preview expiry, output bounds, request-id sanitization).

## Fixed in this PR

Each fix has a regression test in `tests/redteamHardeningP0.test.mts`.

1. **Unicode security normalization** — new `app/lib/security/textNormalize.ts`
   (`normalizeForSecurityScan`: NFKC + zero-width strip + confusable folding).
   Wired into `sanitize.ts` for both injection detection and forbidden-key
   matching, plus broadened injection patterns. (C-01, C-02)
2. **In-memory tenant scoping** — `inMemoryRepositories.ts` and
   `repositoryResolver.ts` now reject cross-tenant `findById`/`findByWorkUnitId`,
   matching D1. (A-2, A-3, B-4)
3. **Preview expiry enforcement** — approval route rejects an expired preview.
   (B-1)
4. **Shared route guards** — new `app/lib/security/routeGuards.ts` centralizes a
   **case-insensitive** `hasClientOwnedFields`, `isPreviewExpired`, and a
   sanitized, length-capped `resolveRequestId`; adopted by the approval,
   action-preview, and tools routes. (C-11, D-6)
5. **LLM output bounds** — `validateLlmOutput.ts` enforces max string length and
   array size. (C-07, C-08)
6. **Provider SSRF guard** — `isSafeProviderBaseUrl` blocks
   private/link-local/metadata hosts and non-HTTPS remotes; enforced in
   `createDeepSeekProviderFromEnv`. (C-06)

## Open recommendations (follow-up, not in this PR)

These need design/schema changes or touch intentional, tested behavior, so they
are tracked separately rather than rushed into a focused hardening PR:

- **Separation of duties (D-2)** — `editor` holding `approve_external_action` is
  currently *intended and tested* (`saasRbac.test.mts`, `lifecycleEndpoints`).
  The correct fix is **four-eyes enforcement** (approver ≠ preview creator),
  which requires persisting `createdByUserId` on the preview (D1 migration) and
  checking it in the approval route. Recommended next.
- **Audit logging (D-1)** — replace the `writeAuditLog` no-op with the
  D1-backed `auditRepo.append` path across all security routes; ensure
  request-id/actor inputs are sanitized (now covered by `resolveRequestId`).
- **JWT issuer/audience (A-1)** — require `iss`/`aud` (fail-closed) in
  production rather than skipping when env vars are unset.
- **Rate-limit client IP (B-2/D-3)** — only trust `X-Forwarded-For` behind a
  known proxy; prefer `CF-Connecting-IP` and ignore client-supplied XFF
  otherwise.
- **GitHub client (D-5)** — `encodeURIComponent` owner/repo before URL
  construction before the real client is wired up.
- **Tenant secret (D-4)** — implement `TenantSecretProvider` so approval hash
  binding is keyed (HMAC) rather than unkeyed SHA-256.
